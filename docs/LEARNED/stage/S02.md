## 问题描述

S02 文献调研中未核实 OMuSense-23 数据集的实际输入输出格式，导致 S03-S06 全部基于错误的数据格式假设进行设计。

具体错误：
- 假设 OMuSense-23 的预处理输出为：胸部位移 + BW（呼吸带宽）+ HW（呼吸谐波宽度）
- 实际 OMuSense-23 的预处理输出为：胸部位移 + BW（呼吸波形 Breathing Waveform）+ HW（心跳波形 Heartbeat Waveform）
- 这意味着输入中已经包含了呼吸波形，任务从"从雷达信号重建呼吸波形"退化为"波形精化/去噪"
- 与用户核心任务"从毫米波雷达信号中提取呼吸特征"严重不符

## 根因分析

1. **文献阅读不深入**：S02 仅阅读了 OMuSense-23 的摘要和概述，未仔细阅读数据预处理章节（§3 Data Format）
2. **术语误解**：BW 和 HW 的缩写被主观臆断为"Breathing Width"和"Harmonic Width"，而实际是"Breathing Waveform"和"Heartbeat Waveform"
3. **缺乏数据集验证步骤**：S02 的方法论/技术方案库（§5）中未包含对数据集格式的详细核实
4. **假设链断裂**：S02 的假设直接传递到 S03-S06，未在中间任何 Stage 进行验证

## 预防措施

### 规则：数据集输入输出格式强制核实

**任何 Stage 涉及数据集选择时，必须执行以下核实步骤：**

```
1. 找到数据集的原始论文/文档
2. 定位数据格式章节（通常是 §3 Data Format 或 §4 Dataset Description）
3. 明确列出：
   - 输入特征的具体名称、维度、物理含义
   - 输出标签的具体名称、维度、类别定义
   - 数据预处理流程（原始信号 → 中间处理 → 最终提供格式）
4. 将输入/输出与任务定义对比，确认是否存在"输入已包含输出"的矛盾
5. 如果存在矛盾，立即停止当前 Stage，报告 Conductor
```

**核实检查清单**：
- [ ] 数据集的缩写/术语是否已确认原始含义？（不能主观臆断）
- [ ] 输入特征是否为原始信号，还是预处理后的特征？
- [ ] 输入中是否已包含输出目标？（防止 trivial 映射）
- [ ] 数据预处理流程是否清晰？（从原始 ADC 到提供格式的完整链路）
- [ ] 数据集的标签定义是否与任务定义一致？

### 规则：缩写术语必须溯源

**遇到数据集中的缩写时，必须**：
1. 在论文中搜索该缩写的首次定义
2. 在数据集文档/代码中确认其实际含义
3. 禁止根据缩写字母主观猜测含义

**示例**：
- ❌ 错误：BW = Breathing Width（主观猜测）
- ✅ 正确：BW = Breathing Waveform（论文原文定义）

## 验证方法

- S02 产出中必须包含"数据集格式核实"子章节
- 每个数据集的文献卡片必须包含"输入输出格式验证"段落
- Conductor 在派发 S03/S06 前，必须检查 S02 中是否已完成数据集格式核实
