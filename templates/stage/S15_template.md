# S15: Ablation Code Generation

> Agent: `experiment`
> Phase: P4
> 目标: 生成消融实验的代码，确保与主实验代码一致

---

## 1. 核心目标

基于 S14 的消融设计和 S11 的主实验代码，生成所有消融变体的实现代码。本阶段的核心任务是：
- **代码一致性**: 消融代码与主实验代码使用相同的基础框架
- **变量隔离**: 确保每个消融只改变目标组件
- **可运行**: 所有消融配置都能直接运行

---

## 2. 消融代码实现方案

| 消融 ID | 变体描述 | 代码修改位置 | 修改方式 |
|---------|---------|-------------|---------|
| Abl-1 (完整方法) | 对照组 | 无 | 使用主实验代码 |
| Abl-2 | 无组件 A | `model.py` 中组件 A 的调用 | 注释掉/替换为 pass-through |
| Abl-3 | 无组件 B | `model.py` 中组件 B 的调用 | 同上 |
| Abl-4 | 简化组件 C | `model.py` 中组件 C 的实现 | 替换为简化版本 |

---

## 3. 配置文件

为每个消融变体生成独立的配置文件：

```yaml
# experiments/configs/ablation_no_component_a.yaml
base_config: main_exp.yaml
ablation:
  id: "Abl-2"
  description: "Remove Component A"
  modifications:
    - file: "model.py"
      action: "comment"
      target: "component_a_forward"
```

---

## 4. 代码修改验证

确保每个消融变体的修改是精确且可逆的：

| 消融 ID | 修改行数 | 影响范围 | 是否只改目标组件 |
|---------|---------|---------|----------------|
| ... | ... | ... | 是/否 |

---

## 5. 与主实验代码的一致性

- [ ] 使用相同的数据加载器
- [ ] 使用相同的评估脚本
- [ ] 使用相同的随机种子策略
- [ ] 使用相同的训练循环
- [ ] 只有目标组件被修改

---

## 6. 验证与检查

- [ ] 代码可运行
- [ ] 与主实验一致
- [ ] 消融变量正确隔离
- [ ] 每个消融有独立的配置文件
- [ ] 修改可逆（通过 git 可恢复）

---

## 7. 风险与限制

| 风险 | 影响 | 缓解措施 | 状态 |
|------|------|---------|------|
| 修改引入连带影响 | 消融不纯粹 | 代码 review，确认只影响目标组件 | 开放 |
| 消融配置与主实验分叉 | 结果不可比 | 继承主配置，只覆盖消融相关字段 | 开放 |

---

- 如果 S16 运行报错 → 回溯到 S15 修复代码

- 如果 S17 发现消融结果异常 → 检查 S15 代码是否正确实现消融
